Using a plain `object` as a lock sentinel has always been a workaround: nothing in the type system signals that the field is a guard rather than data, and accidentally passing a value type to `lock()` silently boxes it each time, creating a fresh object that never actually serializes access. `System.Threading.Lock` fixes both problems by making the locking intent part of the declared type itself — a reader of the code immediately knows what the field is for.

Because the compiler and runtime both understand `Lock` as a first-class synchronization primitive, the JIT can generate more efficient monitor-entry sequences than it can for an opaque `object`. The `EnterScope()` method goes further, returning a ref-struct scope you can hold in a `using` statement, giving you deterministic release without relying on the `lock` keyword at all — useful when the guarded region spans multiple try/finally layers or when you want to make the enter/exit boundary explicit in the code structure.
