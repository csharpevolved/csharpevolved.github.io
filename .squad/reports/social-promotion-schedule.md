## Architecture: Social Media Scheduled Promoter (Azure Function)

### 1. Schedule Data File — `social-schedule.json`

Store the canonical schedule in the GitHub repo at:
```
/social/social-schedule.json
```

Each entry is a flat object. Two entries per week (index 0-based, `"day": "Monday" | "Thursday"`):

```json
[
  {
    "index": 0,
    "week": 1,
    "day": "Monday",
    "isoDate": "2026-07-14",
    "slug": "var",
    "title": "Implicitly typed locals (`var`)",
    "csharpVersion": "3.0",
    "url": "https://csharpevolved.github.io/features/var"
  },
  {
    "index": 1,
    "week": 1,
    "day": "Thursday",
    "isoDate": "2026-07-17",
    "slug": "records",
    "title": "Records",
    "csharpVersion": "9.0",
    "url": "https://csharpevolved.github.io/features/records"
  }
]
```

**43 entries total.** Store the JSON in the repo and expose it as a raw GitHub URL for the Function to fetch, OR commit it to Azure Blob Storage on each repo push via a GitHub Action. The Blob approach is more reliable for production (no GitHub rate limits, no auth needed).

### 2. Azure Function — Timer Trigger

**Project type:** .NET 8 Isolated Worker Azure Functions (C#)
**Trigger:** `TimerTrigger` with CRON `0 0 14 * * MON,THU` (2PM UTC = 10AM ET)

```csharp
[Function("SocialPromoter")]
public async Task Run(
    [TimerTrigger("0 0 14 * * MON,THU")] TimerInfo timer,
    FunctionContext context)
{
    var today = DateOnly.FromDateTime(DateTime.UtcNow);
    var entry = await _scheduleService.GetEntryForDateAsync(today);
    if (entry is null) return; // No post scheduled today

    var posts = await _postGenerator.GenerateAsync(entry);
    await _publisher.PublishLinkedInAsync(posts.LinkedIn);
    await _publisher.PublishXAsync(posts.X);
    await _auditLog.RecordAsync(today, entry, posts);
}
```

**Date resolution logic** in `ScheduleService`:
1. Load the JSON array (from Blob or GitHub raw URL).
2. Find the entry whose `isoDate` matches today's UTC date.
3. If today is past the last scheduled date, use modulo: `entry = schedule[totalDaysSinceStart % schedule.Count]` — automatic cycle restart.

### 3. Post Generation — Two Options

#### Option A: Template-based (simple, reliable)

Pre-authored templates with token substitution. Stored in the repo at `/social/templates/`.

**LinkedIn template** (`linkedin.md`):
```
🚀 C# {Version} brought us {Title} — and it changed how we write {Category} code forever.

{Tagline}

Explore the before/after comparison with real code samples:
👉 {Url}

#CSharp #DotNet #Programming
```

**X template** (`x.md`):
```
C# {Version}: {Title} 🧵
{OneLiner}
See it in action → {Url}
#CSharp #dotnet
```

The `OneLiner` and `Tagline` fields can be pre-authored per feature and stored in the JSON or in the feature's frontmatter in the repo.

#### Option B: Azure OpenAI generation (richer, higher quality)

Call Azure OpenAI GPT-4o with the feature's markdown content as context:

```csharp
var systemPrompt = """
You are Cat Grant, DevRel for csharp-evolved.github.io.
Write a LinkedIn post (150–250 words) promoting the C# feature below.
Tone: professional, enthusiastic, practical. Include a call to action.
End with 3–5 relevant hashtags. No em-dash characters.
""";

var userPrompt = $"""
Feature: {entry.Title} (C# {entry.CsharpVersion})
URL: {entry.Url}
Feature markdown content:
{featureMarkdown}
""";
```

**Recommendation:** Use **Option B for LinkedIn** (long-form, engagement-optimized) and **Option A for X** (character limit demands precision over creativity).

### 4. What the Function Publishes

| Platform | API | Auth |
|----------|-----|------|
| LinkedIn | LinkedIn Marketing API v2 — `POST /ugcPosts` | OAuth 2.0 Client Credentials — store in Azure Key Vault |
| X / Twitter | Twitter API v2 — `POST /2/tweets` | OAuth 1.0a User Context — store in Azure Key Vault |

Both API tokens live in Azure Key Vault, injected via `IConfiguration` / managed identity. Never commit tokens to the repo.

**Audit log:** After each post, write a record to Azure Table Storage:
```
PartitionKey: "2026-07"   RowKey: "2026-07-14"
Slug, Title, LinkedInPostId, XTweetId, GeneratedLinkedInText, GeneratedXText, PublishedAtUtc
```

### 5. Schedule Maintenance Going Forward

| Scenario | Action |
|----------|--------|
| New feature added to site | Add entry to `social-schedule.json` (append or insert) |
| Feature edited/renamed | Update slug/title in JSON |
| Cycle restarts | Function auto-wraps via modulo — no manual intervention |
| Skip a week (holiday) | Add `"skip": true` flag to the JSON entry |
| Regenerate schedule | Run `/scripts/generate-schedule.js` (Node or PowerShell) that reads feature frontmatter and emits a new JSON honoring the version-diversity rule |

The JSON is the **single source of truth.** A GitHub Action should validate the JSON on every PR (`dotnet test` or a simple schema check) to prevent duplicate slugs, version collisions within the same week, or malformed dates.
